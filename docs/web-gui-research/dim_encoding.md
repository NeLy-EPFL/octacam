RECOMMENDATION:
Encode live on the CPU with libx264 ultrafast, true monochrome 4:0:0 (-pix_fmt gray), CRF ~15-18, keyint 1-2 s, one MKV per camera (remux/finalize to MP4 after the trial if downstream tools want it), and keep the per-camera CSV of hardware timestamps. This is the only option VALIDATED on this exact machine: 8 parallel x264-ultrafast-gray streams sustained 150 fps each (1201 fps aggregate) on real rig footage with ~50% estimated headroom once the benchmark's decode overhead is removed, at SSIM-Y ≈ 0.997 and ~25-50x compression — a massive quality/size upgrade over MJPEG at lower CPU cost. Run 8 encoder instances (2-3 threads each) inside the C++/Rust data plane via libavcodec/libx264, fed directly from pylon buffers (no color conversion at all for 4:0:0). Do NOT build around the T400: its NVENC exists and works on driver 470, but measures ~366-700 fps max at 1080p and 2 GB VRAM chokes at ~3 process-level sessions — at best an optional offload for 2 cameras. Skip FFV1 for live capture (~22 cores needed); offer it as an offline archival transcode. Keep a raw-Mono8-to-NVMe mode (O_DIRECT streaming writer; every 50 GB trial fits comfortably inside the KC3000's ~400-500 GB SLC cache) as a selectable high-fidelity mode for special trials, not the default (disk wear: ~66k trials to rated TBW; it's also the OS disk). If more quality headroom is ever needed (superfast/veryfast presets or lossless-ish), the cheapest credible upgrade is an Intel Arc A310/A380 (~$100, no session limits, kernel 6.8 supported) — but bench 8x1080p150 before committing, since no published source confirms that aggregate; single-NVENC NVIDIA cards (T600/T1000/A2000/3050) categorically cannot reach 1200 fps per NVIDIA's own numbers. If browser-native playback of recordings (not just live preview) is a requirement, switch to yuv420p with constant chroma (+33% CPU, still feasible) since 4:0:0 support in browsers/hardware decoders is unreliable.

SUMMARY:
ENCODING STRATEGY FINDINGS (benchmarks run on the actual acquisition machine, i7-12700KF/20T, T400 driver 470.256.02, ffmpeg 4.4, plus primary-source web research).

(1) T400 NVENC — the folklore "T400 has no NVENC" is FALSE, but it doesn't matter. NVIDIA's official support matrix lists T400 with 1x NVENC ("Turing, 6th Gen", H.264+HEVC yes), and I verified empirically on this machine: h264_nvenc encodes fine under driver 470. However, measured throughput at 1080p (best-case all-zero NV12 frames, output discarded): single session preset p4 ≈ 366 fps; preset p1 (fastest) ≈ 700 fps aggregate. Both are far below the 1200 fps target. Worse, the 2 GB VRAM allows only ~3 concurrent ffmpeg NVENC processes — sessions 4-8 fail with "OpenEncodeSessionEx failed: out of memory (10)" (this is VRAM exhaustion from per-process CUDA contexts, not a session-policy limit; T-series pro cards are unrestricted). A single-process/shared-context design would fit more sessions but the ~700 fps chip ceiling stands. Verdict: T400 NVENC can at best offload 2-4 cameras at low quality; do not build around it.

(2) CPU x264 — THE FEASIBLE PATH, validated on real rig footage. A real recording from one of the actual cameras (acA1920-150um fly video, /home/nely/Basler_acA1920-150um__40256533__20240824_183815718.mp4) was used. ffmpeg's libx264 accepts -pix_fmt gray and produces true monochrome "profile High, level 6.1, 4:0:0, 8-bit" (verified in encoder log). Encode-only CPU cost per 1080p gray frame on real content: ultrafast ≈ 5.2 ms, superfast ≈ 13.9 ms, veryfast ≈ 22.3 ms. At 1200 fps aggregate that is ~6.3, ~17, and ~27 core-seconds/s respectively — only ultrafast fits the 12C/20T CPU; superfast does NOT (it is also content-fragile: 7 ms on clean synthetic, 31 ms with synthetic sensor noise). Gray 4:0:0 saves ~25-33% CPU vs yuv420p (yuv420p with neutral chroma from the same gray content: 6.9 ms/frame ultrafast — still feasible if compatibility demands it). End-to-end validation: 8 parallel ffmpeg x264-ultrafast-gray encoders (threads 3 each) on the real footage sustained 150.1 fps per stream / 1201 fps aggregate — and ~40% of that CPU was mpeg4 decode+scale overhead that won't exist when raw frames come from the cameras, implying encode-only aggregate capacity ≈ 1800-2000 fps (~50% headroom at 8x150). Quality/bitrate at ultrafast gray on this content: CRF15 = 44 kB/frame (6.6 MB/s/cam @150fps, ~47:1 vs raw), CRF18 = 21 kB/frame (~99:1, SSIM-Y 0.9970 vs source), CRF20 = 11.8 kB/frame, CRF23 = 4.6 kB/frame (SSIM-Y 0.9965). Caveat: the test source was already mpeg4-compressed (noise-smoothed); raw sensor noise will inflate bitrate (plausibly 2-4x at fixed CRF) and somewhat inflate ultrafast cost (though ultrafast measured the same 5.2 ms/frame on noisy synthetic). Even 4x inflated, 8-camera write rate is ~100-200 MB/s — trivial for the NVMe. For scientific gray video for pose tracking, CRF 15-18 ultrafast is a sensible quality target; dropping to CRF 20-23 still gives SSIM ~0.997 on this content.

(3) FFV1 lossless — NOT feasible live. Measured 18.2 ms CPU/frame (1080p gray, 24 slices); 1200 fps would need ~22 cores; single-instance wall throughput was 262 fps. Measured 12.5:1 ratio is misleadingly good (denoised source); literature/experience for raw noisy mono sensor data is ~2-3:1. FFV1 remains a good OFFLINE archival transcode target.

(4) Hardware-encoder upgrades. NVIDIA's official NVENC performance table (Video Codec SDK 13.0 app note): single NVENC chip at 1080p H.264 p1 CBR = 855 fps (Turing), 868 (Ampere), 910 (Ada); HEVC p1 = 932/943/1055. So NO single-NVENC card (T600, T1000, RTX A2000, RTX 3050 — all 1 chip; T600/T1000 are the same TU117 die as T400) sustains 1200 fps; you'd need dual-NVENC Ada (RTX 4070 Ti+ consumer, or RTX 4500/5000/6000 Ada pro) — not the cheap fix. Consumer session limit is now 8 (raised 3→5 in 2023, →8 in 2024; GTX 1630 excepted) but requires a modern driver. Intel Arc A310/A380 (~$100): no session limit at all, H.264/HEVC/AV1 encode, Linux supported (kernel 6.2+; this machine runs 6.8), and Tom's Hardware measured A380 encoding ~2x a 12900K CPU — promising, but I found NO published number proving 8x1080p150 (1200 fps) aggregate on the single ACM-G11 media block; treat as unverified until bench-tested. All hardware encoders take NV12: feed gray into Y plane + constant-128 chroma plane (write once, reuse) — effectively free, no true 4:0:0 needed.

(5) Raw Mono8 dump + offline transcode — viable as an option, with caveats. 8x150fps x 2.07 MB = 2.49 GB/s; a 20 s trial = ~50 GB. The KC3000 4TB has a pSLC cache absorbing roughly 400-500 GB at ~6 GB/s (Tom's Hardware), so every trial rides the cache easily; post-cache sustained is ~1.0-1.5 GB/s (1TB-model data; 4TB likely higher) — only a risk for very long/back-to-back captures. Endurance: rated 3.2 PB TBW → ~66,000 raw trials; at 50 trials/day that's ~2.4 TB/day ≈ 3.6 years to rated TBW, and the drive is also the OS disk (single partition). 32 GB RAM cannot buffer a full trial (50 GB), so raw mode still needs a 2.5 GB/s streaming writer (O_DIRECT, preallocated). Downsides: disk wear, 48 GB/trial management, mandatory transcode step. Best used selectively (e.g., calibration/golden trials), not as default.

(6) Container — MKV per camera, written live. Plain MP4 writes the moov atom at finalize: a crash loses the whole recording. MKV loses at most the tail and is trivially remuxed afterward (OBS's long-standing default/recommendation: record MKV, optionally auto-remux to MP4). Fragmented MP4 (movflags=frag_keyframe+empty_moov) is the equally-safe alternative if MP4 is a hard requirement. ffmpeg/OpenCV/imageio (hence DeepLabCut/SLEAP) read MKV fine; if a downstream tool insists on .mp4/.avi, lossless remux takes seconds. The 4:0:0 caveat: ffmpeg/libavcodec (and therefore OpenCV/DLC) decode 4:0:0 H.264 fine — proven locally by decoding my own gray encodes through libavcodec — but hardware decoders output fake 4:2:0 or fail, and in-browser playback of 4:0:0 is unverified; for the web GUI, preview should be a separate JPEG/WebRTC tap anyway, so this mostly matters if users double-click recordings in arbitrary players. If that matters, pay the 33% and encode yuv420p with neutral chroma.

(7) Per-frame metadata — keep the CSV sidecar (camera hardware timestamp + frame index + dropped-frame markers); it is the format the analysis ecosystem already consumes and survives any container question. Optionally ALSO stamp container PTS from hardware timestamps (MKV supports arbitrary timestamps; ffmpeg/mkvmerge timecode-v2 ingestion exists) so VFR-aware players show true timing, but do not make the container the only source of truth — many pipelines (OpenCV's CAP_PROP_POS_*, DLC) assume CFR and ignore PTS.
