RECOMMENDATION:
A custom Rust backend is viable end-to-end, with one clearly pragmatic shape: pylon-cxx for acquisition + per-camera ffmpeg subprocesses for encode/mux + axum for the web layer, single static binary deployment. Specifics: (1) Use pylon-cxx; it covers .pfs loading, grab strategies, chunk data, and hardware timestamps today. Software-trigger via the TriggerSoftware command node; replace WaitForFrameTriggerReady with AcquisitionStatus(FrameTriggerWait) polling or upstream a small PR. Pin Pylon 6.x (supported) and verify 7.x linking on the rig before committing; do NOT plan on Pylon 8.1 until issue #24 is fixed. (2) Do not bind an encoder in-process initially: pipe gray rawvideo to one ffmpeg child per camera (libx264 with native -pix_fmt gray High 4:0:0, ultrafast/superfast + CRF ~15–18; h264_nvenc as a switchable option) writing one MP4/MKV per camera. This gets crash isolation, zero unsafe, trivial version pinning, and removes the muxing problem entirely; the in-process upgrade path (rsmpeg, the only well-maintained binding) remains open if process supervision ever becomes the bottleneck. Reject openh264 (≈30 fps/core at 1080p), less-avc (lossless-only, no compression), x264 crate (stale, GPL), rav1e (far too slow) for the main recording path. (3) Web: axum + WebSocket control + MJPEG multipart (or JPEG-over-WS) preview; camera grab loops as std::thread bridged with tokio mpsc blocking_send — all standard, low-risk. (4) Skip the PyO3/FastAPI hybrid: it adds an FFI boundary and dual-runtime packaging without removing any of the hard Rust work; choose it only if Python-extensible online analysis is a hard requirement. (5) Before writing any of this, spend 1–2 days evaluating strawlab's strand-cam (strand-braid repo): it is this exact architecture (pylon-cxx + browser UI + MP4 + openh264/dynloaded-NVENC + in-stream precise timestamps) from the same lab that maintains pylon-cxx — adopting or forking it may collapse the 3–6 week build into integration work. Biggest open risk for any variant is sustained encoder throughput on this specific machine (12700KF CPU x264 capacity vs single Volta-gen NVENC on the T400) — benchmark both at 8x150 fps gray 1080p before locking the design.

SUMMARY:
CAMERA LAYER. pylon-cxx (strawlab) is the only maintained Rust route to Basler pylon. Verified from source (src/lib.rs, src/pylon-cxx-rs.cc): it exposes InstantCamera open/close/start_grabbing/stop_grabbing/retrieve_result(timeout); GrabStrategy enum with OneByOne, LatestImageOnly, LatestImages, UpcomingImage; five node maps (device, TL, stream grabber, event grabber, instant camera) with boolean/integer/float/enum/command nodes; FeaturePersistence via node_map.load/save and load_from_string/save_to_string (CFeaturePersistence::Load under the hood, so .pfs files work); chunk data via grab_result_get_chunk_data_node_map(); and GrabResult.time_stamp() (hardware timestamp) plus block_id(), buffer(), stride(), error info. CRITICAL GAP: InstantCamera::ExecuteSoftwareTrigger and WaitForFrameTriggerReady are NOT bridged — no trigger-related function exists in the C++ bridge. Workaround: software trigger works today via command_node("TriggerSoftware").execute() on the device node map, and trigger-readiness can be polled via AcquisitionStatusSelector=FrameTriggerWait + AcquisitionStatus boolean node (standard on ace cameras); or contribute a small cxx-bridge PR (maintainer accepts API additions — grab strategies were added by community request, issue #17). Maintenance: v0.4.4 (June 2025), issues actively closed through Aug 2025, dual MIT/Apache. VERSION PINNING RISK: Linux build.rs explicitly supports only Pylon 5.x/6.x (panics on other PYLON_VERSION values; default path /opt/pylon); Pylon 8.1 is known broken (all libs renamed with _v3 suffix, open issue #24 since June 2025, plus open issue #29 on version detection). Pylon 7.x may link via the default path since lib names were unchanged, but this is undocumented — must be verified on the acquisition machine. Alternatives: cameleon (pure-Rust GenICam/USB3 Vision over libusb) is real but last released Dec 2024, has no chunk-data parsing (v0.3 roadmap) and unproven at 2.4 GB/s aggregate; aravis-rs (0.11.1, Feb 2025, binds Aravis 0.8.14, GLib-based, auto-generated gir API) is a maintained fallback but adds a whole second vision stack. ENCODING. (a) FFmpeg bindings: rsmpeg (larksuite, 873 stars, FFmpeg 6/7 and an 8.0-tagged release, links via rusty_ffmpeg against system FFmpeg or vcpkg) is the healthiest binding; ffmpeg-next is abandoned, ffmpeg-the-third is its maintained fork (FFmpeg 4.2–8.0). Both give full libavcodec+libavformat (encode gray frames with libx264 and mux in-process) at the cost of unsafe-FFI surface and FFmpeg version coupling. (b) openh264 crate (ralfbiedert): maintained (v0.9), compiles Cisco source via cc, needs I420 input (gray = fill chroma planes with 0x80); its own benchmark shows ~3.9 ms encode at 512x512 on a Ryzen 7950X3D — scaling by area, roughly 30 fps/core at 1080p, so ~1200 fps aggregate is far out of reach on a 12700KF; usable only for preview-rate or low-fps paths. (c) less-avc (strawlab): pure-Rust, zero-unsafe H.264, supports 8-bit/12-bit MONOCHROME 4:0:0 natively — but LOSSLESS-ONLY with PCM-style intra coding, i.e., essentially no compression ratio; fails the "reasonable compression" requirement (fine as an integrity-critical fallback). (d) x264 crate: stale (0.5.0, Dec 2022, Rust 2015 edition) and x264 itself is GPL; note modern libx264 (via FFmpeg) DOES support native monochrome — pix_fmts gray/gray10le (High 4:0:0 profile), which halves work vs faking I420. (e) rav1e: speed-10 1080p runs at tens of fps per instance on desktop CPUs — orders of magnitude short; reject. (f) ffmpeg subprocess per camera (rawvideo gray on stdin): the pragmatic option — crash isolation per camera, zero unsafe, trivially version-pinned by shipping one static ffmpeg binary, encoder+muxer+threading all handled (libx264 -pix_fmt gray, or h264_nvenc), pipe throughput at 311 MB/s/camera is a non-issue (memcpy); cons: 8 child processes to supervise, stderr parsing for errors, container timestamps are CFR unless you remux (hardware timestamps continue to live in CSV — or embed in-stream as strand-cam does: it stores precise timestamps in the H.264 stream metadata). MUXING. Pure-Rust options exist but are fragmented: alfg/mp4-rust (reader+writer, modest maintenance), mp4e (standard + fragmented MP4), muxide (new zero-dep pure-Rust muxer: H.264/H.265/AV1), minimp4.rs bindings, webm crate (libwebm muxer), mkv-element (read/write Matroska elements; matroska/matroska-demuxer are read-only). strand-braid ships a production mp4-writer crate plus ffmpeg-writer. Practical answer: let FFmpeg (subprocess or libavformat via rsmpeg) own the container; consider fragmented MP4 or MKV for power-loss safety. WEB LAYER. axum + tokio is the standard: built-in WebSocket extractor, SPA serving via tower-http; MJPEG preview either as multipart/x-mixed-replace (scottlamb's multipart-stream crate serializes exactly this for hyper; browsers render it in <img>) or JPEG-over-WebSocket to canvas. Grab loops must be dedicated std::thread (allows RT priority/affinity; pylon RetrieveResult blocks); the tokio-documented bridge is tokio::sync::mpsc blocking_send/blocking_recv from the camera threads into async handlers, watch channels for state fanout. This whole stack is an existence proof in strawlab/strand-braid: strand-cam is a Rust app using pylon (runtime-selectable pylon|vimba backend), browser UI (Yew/WASM, built with trunk), MP4 recording with openh264 always available and NVENC optional via a dynamically-loaded libnvidia-encode wrapper (no compile-time SDK needed; relevant for the old 470 driver). Caveat: T400's TU117 carries the Volta-generation NVENC block, a single engine; 1200 fps 1080p aggregate likely exceeds it — needs benchmarking. PYO3 HYBRID. Technically sound: allow_threads + GIL-independent Py<T> lets Rust own real threads while Python (FastAPI) holds the control plane; maturin packaging is mature. But it still requires building the entire Rust data plane AND keeps Python deployment, adds FFI debugging, ABI/wheel management, and cross-boundary lifecycle/shutdown complexity — only worth it if the lab wants Python extensibility hooks (e.g., online analysis). SCALE/RISK. A focused grab→encode→mux→web service for 8 cams is roughly 4–7k LOC Rust (camera manager + trigger pacing ~1–1.5k; ffmpeg-child supervisor ~0.5–1k, or 1.5–2.5k if in-process rsmpeg; axum API/WS/preview ~1–1.5k; state machine/config/CSV ~0.5–1k) plus 1–2k LOC frontend; ~3–6 weeks for a competent Rust dev to reach parity. Risk concentrates in pylon-cxx version pinning (Pylon 7/8), the missing WaitForFrameTriggerReady, and encoder throughput on this CPU/GPU — not in the web or muxing layers, which are commodity. Known ops constraint carries over: ~150 fds/camera means the Rust service must raise RLIMIT_NOFILE just like the Python app.
