RECOMMENDATION:
Use ONE multiplexed binary WebSocket carrying downscaled JPEG preview frames (480x270 grayscale, q70–80, 10–15 Hz, ~10–25 KB/frame, ~10–25 Mbps aggregate) for all 8 cameras, with strand-cam-style per-client drop-stale backpressure (LIFO slot + ack or ws.bufferedAmount). Client decodes with createImageBitmap and blits to a 3×3 grid of <canvas> elements; per-camera flip/rotate via CSS transforms. This is the only option that is simultaneously: single-TCP-port (clean through ssh -L), exempt from the browser's 6-connections-per-host HTTP/1.1 limit (which kills 8× MJPEG multipart or 8× SSE on one origin), encoder-cheap (<1 ms/frame on CPU, zero contention with the recording pipeline), and metadata-capable (camera id, frame number, hw timestamp per message). Reject WebRTC (ICE/UDP/host-IP candidates don't traverse ssh -L; extra encode cost) and WebTransport (QUIC/UDP only; no TCP fallback in browsers). Keep WebCodecs-H.264-over-WS only as a future upgrade — it works on http://localhost (secure context) but adds decoder plumbing for no benefit at preview rates. Control plane: small REST API + JSON status push over the same (or a second) WebSocket; Rust axum + tokio backend with the SPA embedded via rust-embed into the single binary — the exact stack strand-cam validates (axum 0.8). Frontend: build-step-free Lit (vendored, import maps) or plain JS modules; React/Svelte and Yew/WASM are not warranted at this UI scope. Bind to 127.0.0.1 with no app-level auth; SSH is the auth/encryption layer; make the server-side state machine authoritative and broadcast state so multiple tabs are consistent.

SUMMARY:
PREVIEW TRANSPORT — comparison for 8 simultaneous downscaled grayscale streams through one SSH -L TCP port:

(a) MJPEG multipart/x-mixed-replace per camera: simplest (8 <img> tags), and SSH -L itself is not the problem (each TCP connection through the forwarded port becomes its own SSH channel). The killer is the browser: Chrome/Firefox hard-limit HTTP/1.1 to ~6 concurrent connections per host, so 8 always-open multipart streams + REST calls + a status WebSocket starve each other. Escaping the limit needs HTTP/2 (browsers require TLS; h2c unsupported) or multiple ports/origins, which breaks the single-tunnel-port requirement. Also no per-client backpressure, no frame metadata. Reject for 8 cameras on one origin.

(b) JPEG frames over a single binary WebSocket: one TCP connection multiplexes all 8 cameras (small binary header: camera id, frame number, hw timestamp + JPEG payload). WebSockets are exempt from the 6-connection limit (moonfire-nvr explicitly chose WS for this reason; its docs cite a ~256 WS limit). Client side: decode with createImageBitmap() (async, off-main-thread, can run in a worker) and blit to per-camera <canvas>; proof-of-concept projects show ~0.1 s glass-to-glass. Backpressure is easy: per-client drop-stale/LIFO (send newest frame only when the previous send completed or client acked — exactly strand-cam's "firehose" scheme) or watch ws.bufferedAmount. Server cost is trivial: downscale 1920x1080→480x270 + libjpeg-turbo encode of a 130 Kpx grayscale frame is well under 1 ms; 8×15 Hz = 120 encodes/s ≈ a fraction of one core, independent of the recording encoder. Works perfectly through ssh -L. RECOMMENDED.

(c) WebRTC: lowest latency (sub-200 ms–500 ms) but fundamentally hostile to plain SSH tunnels: ICE prefers UDP, host candidates advertise the machine's real interface IPs (unreachable from the tunneled browser), browsers don't offer loopback candidates to remote peers, and TCP candidates/TURN-over-TCP require extra infrastructure; go2rtc, for example, needs a separate 8555/UDP port beyond its API port. It also requires H.264/VP8 encode per stream, eating the limited encode budget (T400, driver 470). Reject.

(d) WebCodecs + WebSocket H.264: technically viable through ssh -L — WebCodecs requires a secure context, but http://localhost qualifies, so no TLS needed; VideoDecoder accepts Annex B H.264 and is HW-accelerated (this is how scrcpy-web/Tango and VibeNVR work). But "reuse the recording encoder's stream" is a trap: recordings are full-res/full-fps (8×1080p decode in the browser), and preview must also work when not recording, so you'd run a second downscaled encoder anyway. Add keyframe-on-join handling, decoder config plumbing, and known Firefox H.264 VideoDecoder bugs. Not worth it for 10–30 Hz grayscale thumbnails; keep as an upgrade path. WebTransport is out entirely: it requires HTTP/3/QUIC over UDP and the spec's HTTP/2-TCP fallback is implemented by neither Chrome nor Firefox — it cannot traverse a TCP-only ssh -L.

PRIOR ART. strand-cam (strawlab/strand-braid, the repo the user linked sibling to pylon-cxx): backend is Rust axum 0.8 + tokio/hyper/tower; live preview is JPEG (quality 80) base64-encoded as a data: URL pushed over Server-Sent Events (EventSource), with ack-based backpressure: per-sender ready_to_send flag, client callback resets it, LIFO slot so stale frames are replaced (web/strand-http-video-streaming/src/lib.rs). Frontend is Yew (Rust→WASM, Trunk build, scss). Auth is a pre-shared token in the URL query (?token=...) exchanged for a session cookie; plain HTTP, trusted-network assumption (web/strand-bui-backend-session). Note strand-cam is one camera per process, so its one-SSE-connection design doesn't hit the 6-connection wall; copying SSE per-camera for 8 cameras would, and base64 adds +33% bandwidth — so adopt its backpressure idea but on one binary WS. moonfire-nvr: live view = WebSocket messages each carrying an fMP4 media segment fed to Media Source Extensions, first message starts with an IDR frame. go2rtc: its video-rtc.js player auto-negotiates WebRTC→MSE over one /api/ws signaling endpoint; WebRTC needs the extra UDP port. frigate: prefers MSE (~0.5–1 s latency, "rock solid"), falls back to jsmpeg (MPEG1-over-WS decoded in JS, CPU-heavy), WebRTC optional with extra port config. The MSE/fMP4 route needs H.264 encode + muxing — unnecessary at preview rates.

BANDWIDTH (measured locally with cv2.imencode on synthetic fly-arena-like frames, /home/nely/octacam-python/.venv python): 480x270 grayscale JPEG — q60: 6.9 KB, q70: 9.6 KB, q80: 13.8 KB, q90: 27 KB; heavy-sensor-noise worst case q70: 25 KB, q80: 35 KB; 640x360 q70: 15 KB. So 8 cams × 15 Hz × ~10 KB ≈ 1.2 MB/s ≈ 10 Mbps typical; absolute worst case (noisy, q80, 30 Hz) ≈ 8.4 MB/s ≈ 67 Mbps. Comfortable on a LAN SSH tunnel; at 15 Hz/q70 even fine over decent WAN. SSE/base64 would inflate this by 33%; binary WS does not.

CONTROL PLANE: REST for request/response (start/stop recording, set fps/duration/save path, trigger source, serial/stepper commands, config) + WebSocket JSON push for telemetry (per-camera fps, drop counters, recording state, disk free) is the pattern all of frigate/go2rtc/moonfire use. You can even share the single preview WS (text frames = JSON telemetry, binary frames = JPEG) or use two WS endpoints — both far under connection limits. Single-binary serving of the SPA is a solved problem in Rust: axum + rust-embed (or static-serve/spa-rs crates) embeds the static dir at compile time (axum discussion #1309 and multiple writeups). This stacks cleanly with the strawlab Rust direction (pylon-cxx). C++ equivalents (uWebSockets, Drogon) exist but the ecosystem and embedding story are weaker.

FRONTEND: the UI surface is small — 3×3 canvas grid, per-camera flip/rotate (pure CSS transforms on the <canvas>, GPU-composited, zero server cost), record button, fps/duration/path forms, per-camera counters, trigger toggle, serial controls. A build-step-free stack is sufficient: Lit via CDN/import maps explicitly supports no-build usage (lit.dev), or preact+htm, or plain vanilla JS modules — vendored locally and embedded in the binary so the lab machine needs no internet/CDN. React/Svelte (build pipeline, node toolchain) is not warranted at this scope. Strand-cam's Yew/WASM route works but imposes Trunk/wasm toolchain and slower iteration; only choose it if you want 100% Rust including shared types.

AUTH/MULTI-USER: bind to 127.0.0.1 only and let SSH provide authentication + encryption — standard for lab tools; strand-cam's token-in-URL→cookie scheme is the fallback model if LAN binding is ever needed. Design single-operator: server-side recording state machine is the source of truth, broadcast all state changes over the WS so any number of viewer tabs stay consistent (multi-viewer is free; conflicting control is resolved last-write-wins).

OPEN QUESTIONS:
Is remote preview ever over high-latency/low-bandwidth links (home/VPN/WAN), or always campus LAN? If WAN matters, add a per-client quality/fps knob or adaptive JPEG quality keyed to ws.bufferedAmount.
Will the recording pipeline land on H.264/NVENC (T400)? If yes and full-res low-latency preview-while-recording later becomes a requirement, the WebCodecs-over-WS upgrade path gets more attractive — verify T400 (driver 470) NVENC session limits in the encoding dimension.
How many simultaneous viewer tabs/users are realistic? Broadcast of the same JPEG buffer to N clients is free, but per-client backpressure state and the 'who can press record' policy should be decided (recommend: server state machine + last-write-wins, no locking).
Which browsers must be supported? The JPEG/WS path is universal; if Firefox/Safari must work, this further argues against WebCodecs H.264 (known Firefox VideoDecoder bugs as of 2024-2025).
Does the preview need overlays (ROI boxes, detected fly positions, saturation/focus indicators)? Canvas blitting makes overlays trivial, but if yes, frame metadata schema on the WS should reserve space for annotation payloads (strand-cam's ToClient struct includes an annotations field worth imitating).
