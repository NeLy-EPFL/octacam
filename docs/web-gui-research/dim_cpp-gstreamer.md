RECOMMENDATION:
GStreamer is a net win only as a per-camera encode+mux engine behind appsrc — not as the camera front-end. Recommended shape for the C++/GStreamer dimension: keep the proven grab/trigger layer (pypylon today, or pylon C++/pylon-cxx if the rewrite proceeds) doing software triggering, hardware-timestamp CSV, and drop accounting exactly as now, and push each camera's GRAY8 frames into `appsrc ! x264enc (4:0:0, superfast/ultrafast, CRF~15-18) ! h264parse ! matroskamux/mp4mux ! filesink`, optionally moving 3-4 cameras to `nvh264enc` (after GRAY8→NV12) to share load with the T400. Do NOT build the system around pylonsrc: it cannot execute software triggers (issue #58 open since 2023), its release cadence is near-dormant (one release since Aug 2024, no pylon 8), and forking an officially-unsupported plugin to add trigger signals erases its convenience advantage. If the team is willing to add an Arduino/triggerbox hardware trigger on Line1 (better cross-camera sync for 3D pose anyway), a stock `pylonsrc`-based pipeline becomes viable and GstPylonMeta (timestamp/BlockID/SkippedImages via pad probe) cleanly reproduces the CSV+drop bookkeeping — that is the only configuration where full-GStreamer is clearly attractive. Among the alternatives researched: plain pylon C++ + libav is a modest delta on the existing cpp/ app (encoder swap is ~300 lines; the web GUI is the real work); Aravis should be rejected; and a Rust hybrid (pylon-cxx + gstreamer-rs or plain ffmpeg) is validated by strawlab's actively-developed strand-cam/braid — which is close enough to the target product (Rust, browser GUI, Basler pylon, H.264/MP4, multi-camera hardware-trigger sync) that evaluating adoption-instead-of-build should precede any rewrite. Hardware caveat that dominates all routes: the T400's Volta NVENC (~500-700 fps 1080p) cannot carry 1200 fps aggregate alone, and 12700KF x264 at 1200 fps aggregate is plausible-but-unproven — benchmark x264 GRAY8 on the rig (extend benchmarks/bench_pipeline.py with a pyav-h264-gray / ffmpeg-subprocess writer) before committing to any architecture.

SUMMARY:
## (1) Basler's official gst-plugin-pylon

Official Basler plugin (BSD-3), but explicitly offered with "no technical support" — issues only. Latest release v1.0.0 (Aug 16, 2024); supports pylon 7.5/7.4 on Linux x86_64 (6.2 only on aarch64). Pylon 8 support exists only in a feature branch (Mar 2025), never released. Activity 2025–2026 is minimal: ~3 commits (last push Feb 16, 2026, a caps fix); 49 open issues, 61 stars.

Capabilities relevant to octacam:
- GenICam features are dynamically exposed as `cam::<Feature>` / `cam::<Feature>-<Selector>` properties (e.g. `cam::TriggerSource-FrameStart=Line1 cam::TriggerMode-FrameStart=On`, `cam::ExposureTime`); stream-grabber params via `stream::` (MaxNumBuffer, MaxTransferSize).
- `pfs-location` property loads .pfs files — matches octacam's existing per-camera `configs/*.pfs` workflow.
- GstPylonMeta is attached to every buffer: camera hardware timestamp, BlockID, ImageNumber, SkippedImages, OffsetX/Y, plus chunk data via `ChunkEnable-*`. Accessible from C/C++ via `gstpylonmeta.h`, from Python pad probes via the bundled `pygstpylon`; sample `show_meta.c`. SkippedImages + ImageNumber give per-frame drop accounting; a pad probe can reproduce the existing timestamps/dropped CSV exactly.
- `capture-error` = keep/skip/abort (default abort; skip aborts after 100 skipped buffers).
- Multi-camera: select by serial/user-name/index; 8 pylonsrc elements in one process is just 8 InstantCameras on the same pylon SDK, so the known ~150 fds/camera exhaustion applies identically (ulimit -n fix, per cpp/README.md and MEMORY.md).

CRITICAL GAP: pylonsrc cannot execute GenICam commands. Feature request basler/gst-plugin-pylon#58 (expose TriggerSoftware/command execution via a signal or device pointer) has been open since Jul 2023 with no implementation. octacam's core recording mode — host software-trigger loop pacing 8 cameras (cpp/src/camera.cpp `ExecuteSoftwareTrigger`, and the Python port) — cannot be reproduced with stock pylonsrc. Options: free-run with in-camera `AcquisitionFrameRate` (smooth pacing but loses cross-camera sync), hardware trigger on Line1 (fully supported via cam:: props, needs a trigger box/Arduino), or forking the plugin to add an action signal. Other maturity signals: issue #30 (corrupted concurrent frames at 1440x1080@200fps) open; #52 (rare startup failure) open.

## (2) Encoding pipeline

- x264enc accepts GRAY8 directly (sink caps: Y444, Y42B, I420, YV12, NV12, GRAY8, 10-bit variants) → true monochrome 4:0:0 H.264 with no videoconvert and no chroma cost: `pylonsrc/appsrc ! video/x-raw,format=GRAY8 ! x264enc ! h264parse ! matroskamux/mp4mux`. Caveat (analysis): 4:0:0 H.264 decodes fine in ffmpeg-based pose pipelines but browsers generally won't play it — keep preview separate, or encode I420 with constant chroma if direct browser playback of recordings matters.
- openh264enc: I420 only, baseline-class quality — not attractive.
- nvh264enc: no GRAY8 (NV12/Y444/RGBA...) — NVENC path needs GRAY8→NV12 expansion (cheap, but +50% data and chroma encode work).
- NVENC capacity on the T400 is the hard limit for GPU offload: TU117 carries a Volta-generation NVENC (no Turing speedup, no HEVC B-frames, single engine). NVIDIA's app note: 1080p H.264 P1 ≈ 667 fps on Pascal, 855 fps Turing; Volta = Pascal × clock ratio. Realistic T400 ceiling ≈ 500–700 fps 1080p H.264 at fastest preset — i.e. roughly half the 1200 fps aggregate worst case. Fine for ~4 cameras at 150 fps or 8 at ≤80–90 fps. Pro card = no fixed session cap, but 2 GB VRAM with 8 sessions is tight, and driver 470 caps you at Video Codec SDK ≤11.1 APIs (newer distro GStreamer nvcodec may want newer drivers — verify).
- CPU x264: published ultrafast 1080p numbers (i7-8700K ≈ 145–200 fps on complex game content; pts/x264-medium ≈ 50–105 fps on 12900K/5950X) say 1200 fps aggregate on the 12700KF is plausible only at ultrafast/superfast with grayscale 4:0:0 and mostly-static fly-arena content — unproven, must be benchmarked on the rig. The repo's own Phase-0 benchmark already proved encode capacity (not Python/GIL) is the bottleneck and that software-trigger pacing rescues MJPEG; the same pacing benefit applies to x264. Likely the right production config is mixed: some cameras on NVENC + rest on x264, or all-CPU at ≤120 fps.
- appsrc/appsink escape hatches are standard and first-class: you can keep the existing pypylon (or pylon C++/Rust) grab loop with software triggering and CSV bookkeeping untouched, and use GStreamer purely as encode+mux per camera (`appsrc ! x264enc ! mp4mux ! filesink`).

## (3) Control plane

- Preview: simplest is multipart-JPEG (MJPEG over HTTP) at ~20–30 fps per camera from any web framework — no GStreamer required. If WebRTC preview is wanted, webrtcsink in gst-plugins-rs is actively maintained (0.15.x, 2025/2026) with a built-in signalling server, `run-web-server=true`, and a JS API.
- Embedding: Python gi/GObject works fine (control-plane perf is irrelevant; pygstpylon exists). Rust gstreamer-rs is first-class and maintained by the GStreamer project itself (crate `gstreamer` 0.25.2, May 2026, 8.3M downloads). A C++ web server (uWebSockets/Crow/Drogon) adds work with no data-plane benefit since encoding happens in x264/NVENC regardless.
- Strong precedent for the Rust route: strawlab's strand-braid (the same group behind the user-linked pylon-cxx) is exactly this product: Rust, browser-based GUI, Basler pylon backend (runtime shim, default pylon 7.3.0), H.264→MP4 recording, multi-camera "Braid" synchronizing cameras via hardware trigger or PTP. Very active: 1.0.0-rc.2 (Jun 2, 2026), pushed daily. pylon-cxx itself: 0.4.4 (Jun 23, 2025), alive but low-traffic (39 stars).

## (4) Plain pylon C++ + libav (no GStreamer)

The existing cpp/ app (~2,500 lines, Qt6) already contains the whole data plane: per-camera grab threads, 150 Hz software-trigger timer, bounded-queue writer (video_writer.cpp, 73 lines), hardware-timestamp CSV + per-index drop tracking, .pfs loading. The rewrite delta is: (a) replace OpencvVideoWriter with a libav writer (libx264 supports `gray` pix_fmt → same 4:0:0 output; ~200–300 lines) or even keep cv::VideoWriter and pipe raw GRAY8 to an ffmpeg subprocess per camera (the "campy" pattern used widely in neuroscience rigs); (b) replace the 900-line Qt MainWindow with an embedded HTTP server + REST + MJPEG preview + small SPA. The encoder swap is small; the web GUI is the bulk of the work in any C++ plan.

## (5) Aravis

Active project (0.8.36, Apr 26, 2026; LGPL; aravissrc GStreamer element), but: USB3 Vision goes through libusb without Basler's tuned stack; historical reports show large performance gaps vs pylon on USB3 (issue #323: 44 fps/66 MB/s Aravis vs 200 fps/300 MB/s pylon on the same hardware — old and ARM-based, but cautionary); no found precedent at 8×311 MB/s ≈ 2.4 GB/s; .pfs files are pylon-specific. Since pylon is free with Basler cameras and already works on this rig, Aravis only adds risk. Not recommended.

## (6) Local grounding

benchmarks/README.md (Phase 0, 2026-06-11) establishes: grabbing 8×150 fps in Python threads is not GIL-limited; OpenCV MJPEG encode capacity is the bottleneck in any language (per-camera processes drop identically); software-trigger pacing brings drops to ~0–0.5%. The "data plane must be C++/Rust" premise is therefore mostly false — the data plane's hot 95% is the encoder library.
