THE STANDOUT MATCH — strawlab Strand Camera / Braid (strand-braid). The user already linked pylon-cxx, which is part of this very project (Andrew Straw's lab). strand-braid is a Rust monorepo (96.5% Rust, dual Apache-2.0/MIT, ~74 stars, ~3,195 commits, 18 releases, actively developed). Strand Camera = single-camera low-latency acquisition + browser-based web UI for remote control; Braid = synchronizes multiple Strand Cameras for multi-camera 3D tracking. Camera backends: Basler pylon (via pylon-cxx, the linked crate), FLIR, and Vimba/Allied Vision. Encoding: H.264 via NVIDIA NVENC, ffmpeg (NVENC/Intel QuickSync/AMD AMF/software), VP8/VP9, plus strawlab's own less-avc; records compressed video to MP4 (and MKV) with precise timestamps; grayscale supported. It has a RAM ring-buffer "time travel" post-trigger (records frames from before the trigger). This is essentially a working open-source implementation of the user's target architecture (Rust data plane + Basler via pylon-cxx + web UI + NVENC MP4). It is tracking-oriented, but the camera/encoding/web crates are directly reusable. STRONGEST candidate to fork or mine.

less-avc (strawlab) — directly answers the grayscale-encoding requirement. Pure Rust H.264 encoder, no unsafe, lossless, dual Apache/MIT. Crucially supports MONOCHROME 4:0:0 at 8-bit and 12-bit (i.e., Mono8/Mono12 natively, no chroma waste) plus 4:2:0 8/12-bit. Lossless = excellent quality but modest compression (intra-only PCM/All-Intra), so it is the "archival/lossless grayscale" option; for "reasonable compression ratio" the user would still pair it with NVENC H.264/H.265 lossy. Good evidence the grayscale-native path is solved in Rust already.

campy (ksseverson57) — Python, MIT. Proven Basler(pypylon)+FLIR(PySpin)+emulation pipeline. Architecture = multiprocessing, one process per camera, raw frames piped to an ffmpeg subprocess (rawvideo over stdin). Encoding via ffmpeg: h264/h265 with NVENC (h264_nvenc/hevc_nvenc), Intel QSV, AMD AMF, or CPU libx264; real-time compression + debayering; output MP4 (config sets quality 0-55, gpuID, fps, dims). Trigger: Arduino/Teensy over serial (115200) for hardware or software/external trigger. Saves frametimes.mat/.npy + metadata.csv. No web UI. Recommends ">1 core per camera." Last real release v2.0 (Jul 2021); issues still opened 2025-2026 but effectively in maintenance. Excellent REFERENCE for the "pipe raw to ffmpeg NVENC, process-per-camera" pattern — the exact upgrade over the user's current cv2.VideoWriter MJPEG bottleneck — but stale and GUI-less.

dattalab-6-cam/multicamera_acquisition — Python; the most battle-tested in real neuroscience labs (the Datta lab 6-camera Basler arena behind keypoint-MoSeq, Weinreb et al., Nature Methods 2024). Supports Basler (pypylon, up to 150 Hz), FLIR, and Azure Kinect (pyk4a). Architecture: separate capture and writer processes per camera with multiprocessing.Queue buffering; queue depth recorded in metadata. Encoding: NVIDIA VPF (Video Processing Framework / PyNvVideoCodec) as primary for max throughput, ffmpeg fallback; AVI container; needs a high-end CUDA GPU (docs cite a GeForce 4080). Trigger: Teensy 4.1 + custom PCB driving hardware trigger and IR lights. Sync files: triggerdata.csv (GPIO transitions, microsecond) + {camera}.metadata.csv (frame_id, camera timestamp, queue_size, line_status) for post-hoc alignment of dropped frames. v1.0.0 (Mar 2024), ~444 commits. No web UI; heavy install; tightly coupled to their Teensy/PCB and Azure Kinect. Best REFERENCE for queue/process model + sync-CSV/dropped-frame bookkeeping at exactly 6x Basler @150Hz.

JARVIS-AcquisitionTool (JARVIS-MoCap, German Primate Center) — C++/Qt/CMake, Windows + Ubuntu 18.04/20.04 + Arch. FLIR only (Spinnaker; v2.7.0 buggy, use <=2.6.0) — NO Basler. Encoding = online JPEG (MJPEG) with optional CUDA GPU acceleration. Arduino Uno trigger. Desktop GUI, no web UI. Verdict: it uses the very MJPEG approach the user is trying to escape and lacks Basler support — NOT worth adopting, though its Qt UX is reference-grade.

Motif (loopbio) — commercial/closed; the gold-standard FEATURE template. Supports Basler/FLIR/Ximea/USB3. Storage: raw, lossless, or industry H.264/H.265 real-time compression. Web-based UI accessible from any browser + HTTP REST API + Python library (python-motifapi) for scripting/scheduling. Quoted throughput e.g. 1920x1200@155fps, 800x600@500fps. Multi-camera + multiple linked Motif boxes; unlimited duration; I/O for stimuli. Motif 5 adds <1ms realtime image streaming for closed-loop without disturbing recording/compression. Can't adopt (closed), but is the exact target UX (web UI + REST + H264/H265/lossless + low-latency live preview). Worth pricing as a buy-vs-build datapoint.

NVR-world architecture lessons (not adoptable apps): moonfire-nvr (Rust) stores cameras' already-encoded H.264 to a sample-file directory + SQLite metadata DB, never decoding/re-encoding (6x1080p30 <10% CPU on a Pi2) — lesson: separate metadata DB from segmented sample files; but our cameras emit raw Mono8 so WE must encode. frigate/go2rtc — for the browser live preview, use go2rtc-style MSE/WebRTC (or simple MJPEG/jsmpeg) to stream a low-fps preview without touching the recording path (mirrors Motif's separate streaming path).

Poor fits: Bonsai/bonsai-rx (C#/.NET reactive node-graph; bonsai-rx/pylon exists; historically Windows-centric, .NET6 cross-platform still maturing; great for prototyping/closed-loop, wrong for headless-Linux+web-UI+max-throughput). Autopilot (distributed Raspberry-Pi swarms, not single-box industrial USB3). ZebVR/closed-loop fish rigs: bespoke, not general acquisition.
