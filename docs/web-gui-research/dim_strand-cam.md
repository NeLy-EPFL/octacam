RECOMMENDATION:
USE AS REFERENCE + PARTIAL ADOPT of crates; do NOT fork the whole strand-braid, and do not build a green-field FFI layer. Concretely: (1) Reuse strawlab's permissively-licensed crates rather than rewriting them — `pylon-shimload` (runtime-loaded Basler binding; avoids build-time pylon SDK and the FFI work the user was about to take on with pylon-cxx) or pylon-cxx if you prefer build-time linking; plus `mp4-writer`, `less-avc`/`less-avc-wrapper`, `openh264`, and the `nvenc`/`dynlink-*` crates for the saving path. These give you .pfs loading, software trigger, hardware + chunk timestamps, and a multi-codec MP4 pipeline essentially for free. (2) Copy the proven UI/transport pattern: axum + token auth + Yew/WASM frontend + JPEG-over-SSE "firehose" for live preview. (3) Write a FOCUSED single-process Rust service that owns all 8 cameras (8 grab threads multiplexed via the fd-based WaitObjects), exposes ONE web GUI, and writes one MP4 per camera — this is less work and a better fit than bending Braid (a 3D-tracking coordinator with a hardware-triggerbox assumption) into a recording-only role, and it lets you share a single NVENC/encoder context. (4) Separately and urgently, validate the encoder budget: adopting strand-cam does NOT solve 8x1080p@150fps on one Turing T400 NVENC. Benchmark T400 H.264 NVENC aggregate throughput first; plan to fall back to less-avc lossless or openh264 on the 12700KF, reduce fps, or add a second/newer NVENC GPU. The license is permissive so any of these paths (fork, vendor crates, reference) are legally fine for an academic lab. Bottom line: strand-cam is the best reference and crate source available and should be mined heavily, but a thin purpose-built recording service that reuses its crates is likely less total work than productionizing braid for pure synchronized recording.

SUMMARY:
strand-braid is a Rust monorepo (github.com/strawlab/strand-braid, dual MIT OR Apache-2.0) housing two apps: Strand Camera (strand-cam, single-camera acquisition + realtime 2D tracking with browser UI) and Braid (braid-run, the multi-camera "mainbrain" coordinator for 3D tracking). Led by Prof. Andrew Straw, Univ. Freiburg. It is EXTREMELY actively maintained: commits dated 2026-06-12 (the day of this research), release 1.0.0-rc.2 on 2026-06-02 and rc.1 on 2026-04-27. (Notably, recent commit messages credit "claude-fable-5" / "Co-Authored-By: Claude Fable 5" — the maintainer is using Claude itself.)

(1) ARCHITECTURE. Backend ~96% Rust. Web server is axum + hyper + tower with token auth (axum-token-auth). Frontend is Yew (Rust compiled to WASM), built with Trunk (`trunk build` in strand-cam/yew_frontend); the dist bundle is embedded into the executable via include_dir/tower-serve-static (feature `bundle_files`). LIVE PREVIEW is a "firehose" task in web/strand-http-video-streaming: each preview frame is JPEG-encoded server-side (convert_image EncoderOptions::Jpeg(80)), base64-wrapped as a data:image/jpeg URL, and pushed to the browser over HTTP Server-Sent Events (SSE, event-stream-types). It is NOT WebRTC/H.264-in-browser — preview is throttled JPEG snapshots, fine for monitoring, not full-rate. Remote control over an SSH tunnel works because it is just a browser-to-HTTP-server.

(2) CAMERAS / MULTI-CAM. strand-cam supports Basler (pylon) and Allied Vision (Vimba X) USB3 + GigE, plus v4l webcams; backend chosen at runtime via `--camera-backend pylon|vimba`. IMPORTANT: strand-braid no longer uses pylon-cxx for its Basler path — it now depends on `pylon-shimload` (camera/ci2-pylon -> pylon-shimload 0.1.1, a brand-new crate published 2026-05-29). pylon-shimload is "derived from pylon-cxx" but dlopens the pylon SDK at runtime through a C-ABI shim, so the pylon SDK is NOT needed at build time (drivers needed only to actually open a camera). ci2-pylon also pulls `rlimit` on Linux — i.e. they explicitly raise RLIMIT_NOFILE, the exact fd-exhaustion problem noted in the user's memory. MULTI-CAMERA MODEL = ONE strand-cam PROCESS PER CAMERA. Braid (braid-run = "mainbrain") reads a TOML with one `[[cameras]]` block per camera and launches/coordinates one strand-cam instance each; cameras may also be "remote" (start_backend="remote") connecting over the network. So 8 cameras = 8 strand-cam processes + 1 braid coordinator, each strand-cam with its own embedded web server/port. Comms: Strand->Braid HTTP (register/updates) + UDP (low-latency 2D features); Braid->Strand HTTP (recording start/stop, frame offsets). SYNC/TRIGGER: braid_types TriggerType is either TriggerboxV1 (a custom external USB "triggerbox" hardware device, e.g. /dev/trig1, with a configured framerate — strawlab's Arduino-based trigger generator giving true hardware sync) or FakeSync (software). Braid only permits recording once all cameras have connected and been synchronized; software-trigger-execute is available per-camera via a GenICam command node.

(3) ENCODERS / CONTAINERS. Native Rust mp4-writer (mp4 crate, MP4 container) with Mp4Codec enum: H264NvEnc (NVIDIA NVENC via dynlink-cuda + dynlink-nvidia-encode, feeds NV12; HEVC GUID present but commented out, only H264 active), H264OpenH264 (Cisco openh264 0.8 software, bitrate/preset rate control), H264LessAvc (less-avc 0.1.4 — pure-Rust LOSSLESS H.264, by Straw; ideal for scientific grayscale since it can store Mono8 luma without chroma-subsampling loss), and H264RawStream (passthrough). Separate ffmpeg-writer path shells out to ffmpeg and supports h264_nvenc, libx264 (with -crf/-preset), h264_vaapi, h264_videotoolbox, h264_nvmpi, and arbitrary codec strings (so hevc_nvenc/libx265, VP9, AV1 are reachable via custom args). Also FMF (Fly Movie Format, raw/lossless), UFMF (background-differenced micro-fmf), y4m, TIFF, SRT (timestamp subtitles); MKV is read (mkv-strand-reader) but saving is MP4. RecordingFrameRate dropdown enumerates up to Fps100 then "Unlimited" (no cap), so 150 fps is allowed via Unlimited. SUSTAINING 1080p@150fps PER CAMERA: per-camera NVENC works for ONE camera, but 8 cameras x 150 fps = ~1200 fps of 1080p H.264 all sharing the single Turing NVENC chip on a T400 (TU117). A single Turing encoder cannot sustain ~1200 fps 1080p H.264; this re-creates the user's existing encoder-capacity bottleneck rather than solving it. less-avc/openh264 fall back to CPU (12700KF) but less-avc lossless produces large files and openh264 at 1200 fps aggregate is also heavy. T400 has 2GB VRAM and old driver 470/CUDA 11.4, a further constraint.

(4) LICENSE. strand-braid, pylon-cxx, and pylon-shimload are ALL dual MIT OR Apache-2.0 (permissive, NOT AGPL). An academic lab can freely fork, modify, and redistribute. The proprietary pylon/Vimba SDKs are separate and loaded at runtime, not bundled.

(5) MAINTENANCE. strand-braid: top-tier active (daily commits through 2026-06-12, marching to 1.0). pylon-cxx: last release 0.4.4 on 2025-06-23, still maintained but now secondary to pylon-shimload; open issues #31 (2026-05, pylon library search), #24 (pylon 8.1 support). pylon-shimload: new (2026-05-29, v0.1.1), the actively-used Basler path.

(6) pylon-cxx API COVERAGE (876-line lib.rs). Present: device enumeration/open/close; start_grabbing with strategy and/or count; retrieve_result; fd-based WaitObject (good for epoll multiplexing 8 cameras); NodeMap load/save of .pfs FeaturePersistence files (node_map_load/save + to/from string) — YES; Boolean/Integer/Float/Enum/Command GenICam node access — YES; software trigger via Command node execute — YES; GrabResult width/height/offset/padding/stride, zero-copy buffer &[u8], block_id, time_stamp (device hardware timestamp) — YES, plus get_chunk_data_node_map for chunk data/chunk timestamps — YES; separate stream-grabber/event-grabber/TL/instant-camera node maps exposed. Missing vs full pylon C++ SDK: no CImageFormatConverter/PylonImage (you convert pixels yourself; strand uses its convert-image crate), no registered event/configuration callback handlers (USB3 events only via polling the event-grabber node map, not C++ callback registration), no CImagePersistence image save, limited high-level chunk parsing. For this use case the coverage is sufficient (pfs load, software trigger, hardware/chunk timestamps all covered).

(7) BUILD COMPLEXITY. Needs rustup + Trunk + a per-app WASM frontend build (strand-cam/yew_frontend, braid/braid-run/braid_frontend). With pylon-shimload, the pylon SDK is NOT required at build time (runtime dlopen); with the older pylon-cxx, the pylon SDK IS required at build (C++ link). NVENC/CUDA are dynamically linked (dynlink-* crates dlopen at runtime), so building without CUDA installed is fine. No single "build everything" script exists due to proprietary deps; .gitlab-ci.yml is the reference and CI builds daily. Reusable crates are published on crates.io (pylon-shimload, less-avc, openh264, nvenc helpers) under MIT/Apache.

(8) ASSESSMENT. strand-cam already implements ~80% of the user's wishlist: Rust backend, browser UI remotable over SSH, Basler USB3 via runtime-loaded pylon, MP4 H.264 with nvenc/openh264/lossless-less-avc, per-camera hardware timestamps, post-trigger ring-buffer ("time travel"), and it even already solves the fd-exhaustion issue (rlimit). It is permissive and intensely maintained. The mismatches: (a) it is one-process-per-camera, and unified 8-camera control comes via Braid, which is a TRACKING coordinator (background subtraction, flydra 3D) oriented around an external hardware triggerbox — heavier than a lab that only wants synchronized RECORDING needs; (b) adopting it does not solve the core 8x1080p150 encoder-throughput problem on a single T400; (c) there is no built-in single-GUI "record 8 cameras to 8 MP4s" mode without either Braid or scripting 8 strand-cam tabs.

OPEN QUESTIONS:
Measured H.264 NVENC throughput of the NVIDIA T400 (TU117, driver 470/CUDA 11.4) at 1080p: how many simultaneous 150fps streams can it sustain before becoming the bottleneck? Needs a direct benchmark.
Can strand-cam run 8 standalone instances (without Braid) under a single unified custom web GUI, or does multi-camera control realistically require running Braid (with its tracking pipeline and triggerbox assumption)?
Does strand-cam/Braid support pure synchronized software-triggered recording WITHOUT the strawlab external hardware triggerbox, at 8x150fps, with acceptable inter-camera sync for behavior analysis (vs needing the Arduino triggerbox)?
Is the JPEG-over-SSE live preview rate-limited enough to not steal CPU/encoder budget from the 8 recording pipelines, and can preview be selectively disabled per camera?
For grayscale fly recordings, what compression ratio vs quality does less-avc (lossless) vs openh264/x264 (lossy) actually deliver on this content, and is lossy H.264 acceptable for downstream pose tracking?
pylon-shimload is brand new (v0.1.1, May 2026) with limited downloads — is it production-stable for an 8-camera lab, or is the older build-time-linked pylon-cxx the safer binding today?
Does the native NVENC path expose HEVC (the GUID is present but commented out), or must HEVC go through the ffmpeg path, for better grayscale compression on the T400?
